Describe the most complex ML model inference you have run. Include details on the compute stack you used and any optimizations you made.

The most complex inference system I have run was a real-time computer vision pipeline for manufacturing monitoring at Conduit. The goal was to infer CNC machine utilization states from production video streams and extract HMI screen text so the system could produce useful machine telemetry rather than just raw model predictions. The inference path combined visual state detection, OCR, edge execution, and structured event publishing into the downstream monitoring pipeline.

The model stack used YOLO-based detection models for machine state recognition and PaddleOCR for CNC HMI text recognition. I benchmarked model variants across six or more operational states and evaluated the tradeoff between accuracy, latency, and robustness under factory conditions such as glare, changing lighting, camera angle variation, and screen reflections. The edge side converted model outputs into structured events and published them through MQTT, which let downstream systems consume machine state and OCR outputs in near real time.

The optimizations were mostly the kind that matter in production: narrowing the label space to operationally meaningful states, comparing model size and inference quality instead of optimizing accuracy alone, tuning OCR for the constrained visual domain of CNC HMI screens, and keeping inference, post-processing, and event transport cleanly separated. That made the pipeline easier to debug and operate, improved production monitoring accuracy by about 45%, and supported an 86.7% OEE monitoring workflow.

What's the most pain you've faced running production systems? What's an interesting change you made to increase resilience or uptime?

The most painful production issues I have faced have come from model-backed systems where failures are intermittent and hard to reproduce. A model can look good in evaluation, but the real product also depends on camera quality, user behavior, device performance, network state, backend synchronization, and the way outputs are presented to users. When those pieces interact, a bad user experience may not obviously be a model bug, an input problem, or an infrastructure problem.

At Wizio AI Physio Coach, I helped build the ML stack for a real-time physiotherapy product, and the important resilience change was to make the pipeline modular instead of treating the model as one opaque endpoint. I worked across preprocessing, model development, testing, proof-of-concept delivery, deployment, and Firebase-backed data flows. Separating those stages made it easier to isolate whether an issue came from input quality, inference behavior, application state, or backend data sync, which reduced the time spent debugging vague production failures.

For the RAG-based chat layer, I focused on grounding and latency as reliability problems, not just model-quality problems. I used a LangChain-powered retrieval approach to reduce hallucinations and processing time by about 70%, which made the feature more dependable for users. The broader lesson I took from that work is that resilient ML systems need clear boundaries around inputs, inference, post-processing, transport, storage, and user feedback. Once those boundaries are explicit, it becomes much easier to add validation, replace weak components, and keep the product usable when one part of the system behaves imperfectly.
